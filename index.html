L0 — Soniox STT

stt-rt-v4 / stt-async-v4 via api.eu.soniox.com
The speech-to-text layer. Listens to Hebrew audio and produces Hebrew text where each word carries a timestamp, a confidence score, and a speaker tag.

Why this layer matters

Without strong Hebrew STT, nothing downstream works. Soniox measured 7.5% WER on Hebrew vs ElevenLabs Scribe v1 at 15.2% and Whisper at 16.1%. Empirically validated on direct test of Hebrew colloquial speech — Soniox was near-perfect, ElevenLabs failed catastrophically.

How it works

  • Streaming: WebSocket at wss://stt-rt.soniox.com/transcribe-websocket
  • Async: HTTPS api.eu.soniox.com/v1/transcriptions
  • Set language_hints=["he","en"] for Hebrew with English code-switching
  • Returns word tokens with start_ms, end_ms, confidence (0–1), speaker
  • Diarization up to 15 speakers via enable_speaker_diarization: true
  • Custom vocabulary via context field, capped at 8,000 tokens — push clinical Hebrew hotwords here

Limits

  • Async max 300 min/file, 10 GB storage, 1,000 stored files
  • Streaming 10 concurrent sessions, 300 min/session, 100 req/min

Cost

$0.10/hr async, $0.12/hr streaming. EU region for PPL Amendment 13 adequacy. DPA pre-signed by Soniox, one-click countersign in Console.

Known gaps

  • N-best alternatives NOT exposed (only top hypothesis). L4 correction must rewrite spans, can't rerank candidates.
  • Confidence score is 0–1 but NOT calibrated for Hebrew specifically. Threshold needs empirical tuning.
  • Hebrew punctuation behavior unverified — the empirical test missed a "?" mark.
Sources: R1 deep dive · Soniox docs · Hebrew page

L4 — Smart text fixes

narrow post-processing on low-confidence spans only
When the STT marks a word or phrase with low confidence, this layer tries to clean it up: adds missing punctuation (especially Hebrew question marks), expands abbreviations, and uses a small Hebrew LLM to rewrite phrases that look wrong. It does NOT retranscribe the whole transcript — only touches the uncertain bits.

Why this layer exists

Soniox is good but not perfect at Hebrew punctuation and clinical jargon. A narrow, surgical fix on uncertain spans can plausibly help — but only if empirical measurement on Hebrew shows it actually does. Text-only LLM rescoring of full transcripts was already measured DEAD on Hebrew (KenLM regressed, DictaLM flat). Narrow correction is a different problem class, but it has the same risk of regression if mis-applied.

Pipeline (cheapest to most expensive, short-circuit on success)

  1. Punctuation restorer — AlephBERT-punct adds "?" "." ","
  2. Abbreviation expander — rules engine + curated dict (gershayim heuristic + lookup)
  3. NER guard — DictaBERT-NER protects recognized entities from over-correction
  4. LLM rewrite on span — DictaLM-3.0-1.7B-Instruct fp16 on SageMaker A10G via vLLM (~$0.0014/1k tokens self-host)
  5. Hard-case fallback — Claude Sonnet API gated to <5% of spans by confidence

Critical gap

ZERO published Hebrew empirical evidence that this stack improves a 7.5% Hebrew baseline. Mandatory: measure span-level edit precision vs harm rate on a held-out Hebrew clinical set before locking L4 in. Decision gate: ship L4 only if precision > harm. (See M-l4-hebrew, R-AAA, §7j.)

Why not text-only LLM rescore of the whole transcript

Already measured DEAD on Hebrew in prior phases. KenLM 5-gram regressed on noisy speech. DictaLM-1.7B perplexity rescoring was flat after proper Hebrew normalization. Full-text rescoring is dead. Narrow span correction is the only L4 path still alive — conditional on measurement.

Sources: R6 narrow correction · DictaLM 3.0 paper

L1 — Trust Mode UI

Abridge Linked Evidence pattern, not confidence heatmap
What the clinician sees on their screen. The Hebrew transcript is shown right-to-left. Click any word and that word's audio plays back. Edit anything that looks wrong. Highlight phrases with clinical notes.

Why this pattern

Trust isn't built by painting words red/yellow/green based on confidence — the CHI '25 study (Kuhn & Kersken) found that confidence heatmaps neither improve correction efficiency nor are perceived useful by users. Trust IS built by letting clinicians verify any word against the source audio in one click (Abridge's "Linked Evidence" pattern, used by every clinical scribe vendor that scales).

Interaction patterns

  • Click any word → audio seeks to that word and plays 2-second pre-roll
  • Streaming UX → finalized words rendered solid + editable; tentative (not-yet-final) words rendered at 60% opacity + READ-ONLY ("prefix-lock" pattern). Prevents lost edits when the recognizer revises.
  • Low-confidence cue → subtle dotted underline only. No color heatmap.
  • Double-tap a word → bottom sheet appears with: play just this word, phonetic suggestions (Hebrew Double-Metaphone), LLM rewrite-in-context, or free text edit.
  • Annotation → select a range → highlight in one of 4 clinical colors (phonology, syntax, pragmatics, fluency) or attach a threaded note. Annotations anchor to word IDs + timestamps so they survive transcript re-edits.
  • Edit-capture loop → every accepted correction logs {audio_span, original, corrected} after on-device PHI scrub. Opt-in default-off for clinical compliance.

Hebrew RTL specifics

  • Transcript container dir="rtl"
  • Audio controls stay LTR (play buttons, timestamps mm:ss)
  • Unicode bidi isolates (U+2068 / U+2069) around English/number runs

Mobile pattern

Scrolling transcript on top, persistent waveform + 48-pt play button at bottom (thumb-zone). Used by Google Recorder; works well for clinical tablet workflows.

Accessibility

WCAG AAA contrast (7:1). Confidence redundantly encoded (underline + opacity, never color alone — ~8% of male users have red-green CVD). Full keyboard navigation. ARIA live region announces newly-finalized streaming text.

Sources: R5 UI patterns · Abridge platform · CHI '25 confidence study

L2 — Clinical SLP analysis

the actual product moat
The clinical-pathology brain of the product. Once you have a clean Hebrew transcript and the raw audio, this layer extracts signals a speech-language pathologist actually cares about: how fast the patient speaks, how often they pause, whether they're missing words, what kinds of errors show up in their speech, and how all of that compares to the same patient's previous session.

Why this is the moat

Anyone can buy Soniox STT. Anyone can build a transcript editor. What's hard — and what we own — is the Hebrew-specific clinical interpretation: which Hebrew phonological patterns indicate which pathology, which Hebrew syntax errors map to which aphasia subtype, and how Hebrew prosody contours diverge in dysarthria. This is where the TAU/Friedmann partnership and our own Hebrew aphasia corpus live.

Feature categories

Acoustic (from raw audio)

  • Jitter, shimmer (voice quality)
  • F0 trajectory (pitch curve)
  • Formant frequencies (F1, F2, F3)
  • Articulation rate (syllables/sec)
  • Pause distribution + duration

Lexical (from transcript)

  • Filler ratio ("אה", "אמ")
  • Anomia gaps — silences before content words
  • Paraphasias — semantic + phonemic substitutions
  • Code-switching rate (Hebrew ↔ English)
  • Type-token ratio, MLU (mean length of utterance)

Session-over-session

  • Per-patient diff vs previous session
  • Trend lines across the treatment arc

Validated Hebrew scales we map to

  • WAB-Hebrew (Kasher et al. 1999)
  • FDA-2 Hebrew for dysarthria (Icht et al. 2022, Ariel University)
  • Friedmann-lab batteries: BAFLA (agrammatism), FriGvi (phonological STM), SHEMESH (100-picture naming), HIFIL (verb-morphology)

Data gap

AphasiaBank has ZERO Hebrew corpus. Verified directly on talkbank.org — the Hebrew language doesn't appear. We build our own corpus using AphasiaBank's standardized protocol: Cinderella retell + 4 picture descriptions + 1 procedural-discourse task, CHAT transcription. Requires clinical partner + IRB + ~50–200 patients.

Academic anchor

TAU Sagol / Friedmann Language & Brain Lab is the primary partnership target. Naama Friedmann is active, holds the Branco Weiss Chair, recent Hebrew aphasia work includes Keret et al. 2023 on Primary Progressive Aphasia in Hebrew speakers. Secondary: Haifa CD department; Ariel (Dr. Michal Icht) for dysarthria.

Sources: R3 SLP ecosystem · Friedmann lab · AphasiaBank corpora

L3 — Compliance + Audit

PPL Amendment 13 + AMAR/AMIR SaMD
What the law requires before an Israeli hospital can put this in front of real patients. Two parallel tracks: privacy law (PPL Amendment 13) and medical device law (AMAR/AMIR). Most of this is paperwork + external certifications, not engineering.

Privacy: PPL Amendment 13 (effective 14 Aug 2025)

  • DPO appointed — external contractor allowed, but cannot be CEO, CTO, IT-Manager (conflict-of-interest disqualifications)
  • DPIA filed — clinical AI is an automatic trigger (three independent reasons: sensitive data + automated decision-making + new technology)
  • Database Definition Document — statutory record listing purposes, categories, recipients, retention, security level (HIGH for health)
  • Patient consent in Hebrew — granular, informed, explicit, with disclosed Soniox EU cross-border transfer + retention + rights + withdrawal mechanism
  • Encryption at rest and in transit
  • Access logs retained ≥24 months (Privacy Protection Data Security Regulations 2017, Reg. 10)
  • Pen-test + risk survey every 18 months — engage Sygnia or Comsec (Israeli firms with healthcare experience)
  • Breach notification: 24-hour target, 72-hour ceiling to PPA; PPA may order notification to affected data subjects

Medical device: AMAR/AMIR

  • SLP transcription + clinical-feature extraction intended for diagnosis/monitoring/treatment of speech disorders qualifies as SaMD under the Medical Equipment Law 5772-2012
  • Likely Class II / IIa (clinical support, non-life-sustaining) — class hinges on intended-use wording (decision support vs diagnostic)
  • Israel uses reference-country class — we register via FDA 510(k) OR CE MDR, then leverage that for AMAR
  • Class I/A with prior reference-country clearance: self-declaration, cert in 48h. Higher classes: full dossier review, 4–9 months
  • Israeli Registration Holder (IRH) required — local entity with ISO 9001

Quality standards (required for Class IIa+)

  • ISO 13485:2016 — medical device QMS
  • IEC 62304 — software lifecycle; SLP analysis software is Safety Class B (non-serious injury possible)
  • No notified body in Israel — engage a CE notified body (BSI, TÜV SÜD)

Insurance

Product liability + professional indemnity, $1–5M per claim typical for clinical SaaS in Israel. Israeli Product Liability Law 5740-1980 imposes strict liability on manufacturers.

Hospital procurement extras

Beyond the legal minimum, Israeli hospitals (Sheba ARC, Hadassah Innovation, Loewenstein) typically require: ISO 27001, SOC 2 Type II, recent pen-test report (<12 months), DPIA copy, AMAR certificate, Hebrew-language UI evidence, IRB/Helsinki approval for any data collection, BAA-equivalent DPA, and indemnity insurance certificate.

Sources: R2 compliance roadmap · IAPP Amendment 13 · Emergo AMAR guide

Architecture v4.0 stack

committed · rev 1

The stack

L0 Soniox STT → L4 narrow correction (measurement-gated) → L1 Trust Mode UI → L2 Clinical SLP analysis → L3 Compliance + Audit.

What this replaced

v3.0 plan was "train our own Hebrew ASR from scratch via SLAM-LLM linear projector + LoRA." That plan empirically failed in Phase 1: 9,000 training steps produced a model that ignored audio entirely (probe confirmed: 4 of 5 different Hebrew audios returned identical hypothesis). WER 614%. The architecture was inadequate for non-English at our data scale.

Key design commitments

  • STT is a commodity; the moat is L1 + L2 + L3, not L0
  • L4 is measurement-gated — we don't ship narrow LLM correction until empirical evidence on Hebrew shows it helps
  • Trust Mode UI uses Linked Evidence (Abridge pattern), not confidence heatmaps (CHI '25 showed heatmaps don't help)
  • Compliance is parallel-tracked from day one, not bolted on at the end

Realistic WER target

5–9% Hebrew, not 3–4%. No open Hebrew ASR hits 3–4% today. Soniox at 7.5% is the best documented baseline. Honest forecast: with L4 helping marginally, 5–8%. Without L4, the Soniox baseline.

SLAM-LLM linear projector ASR

reversed · abandoned

What was attempted

Train a SLAM-LLM linear projector + LoRA r=32 on q_proj/v_proj of DictaLM-3.0-1.7B-Base. Frozen Whisper-large-v3 encoder, frozen LLM, only projector + LoRA trained. 50k Hebrew samples from crowd-transcribe-v5, 9,000 training steps.

What happened

  • Teacher-forcing eval_acc climbed 0.31 → 0.38 across the run (looked like learning)
  • Free-run WER on FLEURS he_il: 614% (generates more text than the reference, all wrong)
  • Audio-routing probe (5 different Hebrew audios + silence + noise): the model produced the EXACT SAME Hebrew hypothesis for 4 of 5 different real audios. Silence/noise → English fallback.
  • Conclusion: the projector learned audio-shape detection ("is this Hebrew speech?"), not audio-content alignment. The LLM learned Hebrew bigram statistics from the right-shifted target during teacher forcing, not from audio.

Why it failed

Linear projector + LoRA is an adequate bridge for English at SLAM-LLM's LibriSpeech scale (~1000+ hours) because the LLM has strong English priors and Whisper features are already English-aligned. For Hebrew at 50k samples, the linear bridge can't carry the burden of teaching the LLM a new modality. This matches the VOX-KRIKRI paper's motivation for adding residual cross-attention on Greek.

What's preserved

Code, scripts, eval pipeline, and the step_9000/model.pt checkpoint are kept locally as a historical artifact. The infrastructure (data prep, eval harness, audio-routing probe) is reusable; the trained checkpoint itself is a paperweight as an ASR.

Was it wasted?

The Vast compute on Phase 0+1 (~$5–15) was wasted on the artifact. The probe technique and the failure-mode finding (M-tf-vs-freerun) are reusable for future architecture decisions.

Empirical: output/slam_phase1_eval_results/smoke_n20_v2.json · output/slam_phase1_eval_results/baseline_n200.json

"Israel residency required" framing

reversed · corrected

What I claimed earlier

That PPL Amendment 13 required Israeli physical data residency for clinical health data. This framing implied Soniox EU couldn't be used and we'd need to self-host ivrit-ai on AWS il-central-1.

What the law actually says

Amendment 13 requires one of three: (1) destination country provides adequate protection (EU qualifies), (2) DPA / standard contractual clauses, OR (3) data subject consent. None of those mandate physical Israeli residency.

What this means

Soniox EU + DPA + patient consent satisfies the law. Health data still requires HIGH-security obligations (encryption, access logs, pen-test) on our side, but the STT provider being EU-based is fine. Self-hosting ivrit-ai is no longer compliance-mandatory — it's now just a backup option if we want a non-API fallback.

Why I drifted to overcautious

Pattern-matched "clinical data" + "AI Israel" to "must keep on Israeli soil" without reading the statute. The user correctly called this out (R-brain — apply judgment / read the docs before asking).

  • Data: 50,000 ivrit-ai/crowd-transcribe-v5 samples
  • Steps: 9,000 of 10,000 (interrupted)
  • Checkpoint size: 90.5 MB (projector ~66 MB + LoRA ~24 MB)

Result

Free-run WER on FLEURS he_il (n=20 smoke): 614%. Audio-routing probe confirmed the model produces the same Hebrew hypothesis for 4 of 5 different real Hebrew audios. The model learned to be a 2-class detector ("is this Hebrew speech?"), not an ASR.

Whisper baseline (for comparison)

Whisper-large-v3 on FLEURS he_il (n=200, same normalization): WER 32.12%, CER 12.19%. Sample outputs are recognizable Hebrew transcriptions with minor word-level errors — a real ASR baseline.

Decision

SLAM-LLM linear projector architecture abandoned for Hebrew. Pivot to using existing strong Hebrew STT (Soniox) and building the moat above the STT layer.
